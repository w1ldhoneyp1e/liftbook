# MVP 2 Plan

Дата: 2026-05-01

Цель MVP 2: ускорить повторяющиеся тренировочные сценарии и сделать ежедневное использование Liftbook заметно удобнее без ухода в “большую платформу”.

Перед активной разработкой новых фич сначала нужно закрыть найденные проблемы из [docs/ru/mvp2-found-issues.md](/home/kirill-yashmetov/projects/liftbook/docs/ru/mvp2-found-issues.md).

## Продуктовый фокус

После MVP 1 у нас уже есть:

- рабочий дневник тренировок;
- offline-first поток;
- guest account и базовый sync;
- backend на своем VPS.

Значит в MVP 2 главный вопрос уже не “можно ли пользоваться”, а:

**как сделать так, чтобы пользователь тратил меньше времени на рутину во время тренировки.**

## Основные цели MVP 2

1. уменьшить повторный ручной ввод;
2. ускорить сборку типичной тренировки;
3. сделать историю упражнений полезнее в момент выполнения;
4. не раздувать продукт в сторону аналитической платформы слишком рано.

## Что входит в MVP 2

## 1. Шаблоны тренировок

Зачем:

- пользователь часто повторяет похожие тренировочные дни;
- сейчас это приходится собирать заново.

Что дать:

- создать шаблон из текущего дня;
- применить шаблон к выбранной дате;
- редактировать шаблон;
- удалить шаблон.

MVP-уровень:

- без сложного шаринга;
- без тренерских сценариев;
- без библиотек шаблонов.

## 2. Переиспользование прошлой тренировки

Зачем:

- это более быстрый сценарий, чем ручное добавление упражнений по одному.

Что дать:

- скопировать набор упражнений и подходов из прошлого дня;
- вставить его в текущую дату;
- при вставке сохранить возможность сразу редактировать.

## 3. История упражнения как рабочий инструмент

Зачем:

- одного “последнего результата” со временем становится мало.

Что дать:

- экран истории упражнения;
- последние несколько выполнений;
- веса/повторения по датам;
- быстрый доступ из карточки упражнения.

Фокус:

- не графики ради графиков;
- сначала практическая история для принятия решения во время тренировки.

## 4. Заметки к упражнению или подходу

Зачем:

- иногда нужно сохранить контекст: техника, боль, темп, пауза, комментарий к сету.

Что дать:

- короткую заметку к упражнению;
- короткую заметку к подходу.

MVP-уровень:

- без rich text;
- без вложений;
- просто текст.

## 5. Расширенная метаинформация подхода

Зачем:

- не все подходы равны;
- часть пользователей хочет различать разминочные и рабочие подходы.

Что дать:

- warm-up;
- working set;
- failure / near failure;
- возможно later: RPE/RIR, но только если не усложнит UX.

Рекомендация:

- в MVP 2 не перегружать это обязательностью;
- сделать это легкой опциональной меткой.

## 6. Расширение каталога упражнений

Зачем:

- 50 упражнений хватило для MVP 1, но дальше каталог быстро начнет упираться.

Что дать:

- движение к ~150 упражнениям;
- нормализация мышечных групп и названий;
- сохранение пользовательских кастомных упражнений как первого класса.

## Что пока не входит в MVP 2

- полноценная аналитика и большие графики;
- multi-device sync “по-взрослому”;
- социальные функции;
- корпоративная интеграция;
- тренерский режим;
- платежи;
- роли и организации.

## Технический фокус MVP 2

Продуктовые фичи выше потребуют нескольких технических опор:

1. доработать domain model под шаблоны;
2. аккуратно расширить sync payloads под notes и set metadata;
3. добавить безопасную server-side эволюцию схемы без поломки существующих guest users;
4. не ломать быстрый mobile-first flow ради “богатых сущностей”.

## Порядок реализации

### Этап 0. Закрыть найденные проблемы

Сначала закрываются пункты из `mvp2-found-issues.md`.

Только после этого имеет смысл идти в новые продуктовые сценарии ниже.

Если делать MVP 2 итерациями, я бы шел так:

1. **История упражнения**
2. **Переиспользование прошлой тренировки**
3. **Шаблоны тренировок**
4. **Заметки**
5. **Метаинформация подходов**
6. **Расширение каталога**

### Почему такой порядок

- история и переиспользование сразу дают прикладную ценность;
- шаблоны становятся понятнее, когда уже есть reuse flow;
- заметки и set metadata лучше добавлять после стабилизации основной UX-модели;
- каталог полезен, но не должен съесть все время раньше core workflow.

## Критерий готовности MVP 2

MVP 2 можно считать собранным, если:

- повторяющаяся тренировка собирается заметно быстрее, чем в MVP 1;
- у пользователя есть рабочий способ опереться на историю упражнения;
- заметки и set metadata не перегружают основной flow;
- продукт остается быстрым в мобильном использовании;
- sync и backend остаются совместимыми с уже существующими данными MVP 1.
