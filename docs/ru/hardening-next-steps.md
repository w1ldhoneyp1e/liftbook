# Hardening: следующие шаги

Дата: 2026-05-01

Документ фиксирует основные эксплуатационные риски Liftbook после первого VPS-релиза и рекомендуемые следующие шаги.

## Текущее состояние

Сейчас Liftbook работает как один VPS-стек:

- `web`
- `api`
- `postgres`
- `caddy`

Релиз автоматизирован через [scripts/release-vps.sh](/home/kirill-yashmetov/projects/liftbook/scripts/release-vps.sh).

Перед каждым релизом создается backup PostgreSQL.

## Что уже хорошо

- есть единый release-flow;
- есть backup перед миграциями;
- есть post-deploy healthcheck;
- есть Docker Compose контур для повторяемого запуска;
- есть явный production env-файл.

## Главные риски

### 1. Один VPS — single point of failure

Если падает сервер или диск, одновременно теряются:

- frontend;
- backend;
- database;
- локальные backup-файлы.

### 2. Backup сейчас локальный, а не offsite

Backup создается на том же сервере, где живет PostgreSQL.

Это значит:

- backup полезен против неудачного релиза;
- backup не спасает от полной потери VPS или диска.

### 3. Backup делается по релизу, а не по расписанию

Сейчас backup создается перед `pnpm vps:release`.

Если между релизами пройдет много времени, то окно потери данных растет.

### 4. Нет готового restore-tool

Backup есть, но восстановление пока ручное. Это лучше, чем ничего, но слабее, чем заранее проверенный restore-flow.

### 5. Нет инфраструктурного мониторинга

Пока нет:

- uptime monitoring;
- alerts по памяти;
- alerts по диску;
- alerts по restart контейнеров;
- внешнего контроля доступности `/api/health`.

### 6. Нет hardening SSH и доступа к серверу

На старте это приемлемо, но позже нужно:

- уйти от `root` как основного рабочего способа входа;
- перейти на SSH-ключи;
- ограничить firewall;
- при необходимости включить fail2ban.

### 7. Нет staging-окружения

Любой релиз идет сразу в production. Для MVP это нормально, но это все еще operational risk.

## Рекомендуемый порядок hardening

## 1. Регулярные backup по cron

Цель:

- делать backup не только при релизе, но и по расписанию.

Рекомендация:

- nightly backup, например раз в сутки ночью.

## 2. Offsite backup

Цель:

- хранить копию backup вне основного VPS.

Примеры:

- S3-compatible storage;
- второй сервер;
- объектное хранилище провайдера.

Это самый важный следующий шаг после release-time backup.

## 3. Restore script + test restore

Цель:

- не только уметь делать dump, но и уметь предсказуемо восстановиться.

Рекомендация:

- сделать отдельный `restore-vps.sh`;
- один раз реально проверить восстановление на тестовом окружении.

## 4. SSH hardening

Цель:

- уменьшить риск компрометации сервера.

Рекомендация:

- создать отдельного deploy-пользователя;
- включить вход по SSH-ключу;
- отключить password login для `root`;
- настроить firewall.

## 5. Базовый мониторинг

Минимально полезно:

- внешний ping/HTTP-check;
- контроль свободного места;
- контроль использования RAM;
- уведомление о падении сервиса.

## 6. Log rotation и housekeeping

Цель:

- не позволить логам и старым артефактам занять диск.

## 7. Позже — staging

Не обязательно прямо сейчас, но полезно после появления более частых релизов.

## Приоритет

Если делать только три следующие вещи, я бы делал так:

1. **Nightly backup**
2. **Offsite backup**
3. **Restore script + test restore**

Это даст наибольший прирост надежности при минимальной дополнительной сложности.
