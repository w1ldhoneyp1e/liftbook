# Черновик доменной модели

## Основные сущности

### User

Пользователь Liftbook. Поддержка аккаунтов нужна рано, но локальная запись тренировок не должна зависеть от сети.

Пользователь может начать как локальный/гостевой и позже привязать аккаунт для backup и sync.

### Exercise

Движение, которое можно добавить в запись тренировки.

Важные поля:

- стабильный id;
- опциональный server id после sync;
- локализованное название;
- группы мышц;
- tracking mode;
- встроенное или пользовательское упражнение;
- sync status;
- timestamp удаления для мягко удаленных пользовательских упражнений.

### Muscle Group

Группа для навигации и фильтрации упражнений.

Начальные примеры:

- Грудь
- Спина
- Ноги
- Плечи
- Бицепс
- Трицепс
- Кор
- Ягодицы
- Все тело
- Кардио
- Другое

Названия должны быть локализованы.

### Workout Record

Запись тренировки пользователя. Не требует строгого события start/finish.

Важные поля:

- user id или local owner id;
- дата тренировки;
- локальное время создания;
- время последнего обновления;
- опциональное время удаления;
- sync status;
- опциональный заголовок;
- опциональные заметки;
- exercise entries;
- опциональная вычисленная длительность.

### Exercise Entry

Одно упражнение внутри записи тренировки.

Одно и то же упражнение может встречаться в одной тренировке несколько раз.

Важные поля:

- exercise id;
- позиция/порядок;
- set entries;
- источник прошлого результата, если дефолты были сгенерированы из истории;
- время последнего обновления;
- опциональное время удаления;
- sync status.

### Set Entry

Запись подхода.

MVP не разделяет planned и completed подходы. Подходы, созданные из прошлого результата, сразу считаются обычными записями и могут быть отредактированы или удалены.

Возможные поля зависят от tracking mode:

- вес;
- единица веса;
- повторения;
- длительность;
- дистанция;
- флаг собственного веса;
- заметки;
- опциональное время удаления.

Set entries вложены в exercise entry в локальной MVP-модели. Редактирование или удаление подхода помечает родительский exercise entry как sync-pending.

### Rest Timer Event

Событие таймера отдыха между подходами.

Такие события могут использоваться для примерной длительности тренировки: от первого старта отдыха до последнего завершения.

### User Settings

Настройки, влияющие на ввод и отображение.

Важные поля:

- предпочитаемая единица веса: kg или lb;
- локаль: сначала ru или en;
- настройка автотаймера отдыха;
- флаг дефолтов из прошлого результата;
- шаг kg, по умолчанию 1 kg;
- шаг lb, точное значение открыто;
- время последнего обновления;
- sync status.

## Sync Metadata

Локальные syncable-сущности должны иметь достаточно метаданных для сверки с backend:

- стабильный локальный id;
- опциональный server id;
- created timestamp;
- updated timestamp;
- опциональный deleted timestamp;
- sync status: pending, synced или conflict.

Удаления для syncable workout data должны быть мягкими. UI сразу скрывает удаленные записи, а sync позже отправляет удаление на backend.

### Sharing Consent

Согласие пользователя на передачу данных внешней или корпоративной интеграции.

Важные поля:

- integration id;
- scope передаваемых данных;
- статус согласия;
- timestamp выдачи согласия;
- timestamp отзыва согласия.

## Правило прошлого результата

Когда пользователь добавляет упражнение, Liftbook ищет самую свежую прошлую запись тренировки с тем же упражнением.

Если запись найдена, Liftbook может заранее создать такое же количество подходов с тем же весом и повторениями как редактируемые дефолты.

Для MVP нужен только один самый свежий прошлый результат.

## Объем стартового каталога

MVP должен включать около 50 встроенных упражнений. После MVP 1 желателен расширенный каталог примерно до 150 упражнений.

Пользовательские упражнения важны и должны попасть в MVP 1, если возможно, или сразу после MVP 1.
